What is Steward ACS?

Infrastructure layer that coordinates AI agents — preventing conflicts, sharing knowledge, and providing structured tooling.

Steward ACS is a standalone coordination system for AI agent fleets. It provides task lifecycle management, distributed file locking, persistent knowledge memory, and an MCP tool gateway — all accessible via simple tool calls that agents can invoke directly.

Think of it as air traffic control for AI agents. Each agent registers its presence, claims work, locks resources, and shares learnings. The system prevents duplicate work, resolves conflicts, and preserves institutional knowledge across sessions.

Built with Elixir/Phoenix and running on port 4001, it exposes a comprehensive set of acs_* MCP tools that any AI agent can call — no SDKs, no libraries, no vendor lock-in.

Steward ACS runs as a standalone Phoenix web application (port 4001). It exposes MCP tools (acs_* prefix) that AI agents call directly. The system uses ETS for fast in-memory caching with a PostgreSQL (production) or SQLite3 (development) database for persistence. A background sweeper handles auto-release of stale tasks and locks. The MCP Gateway routes agent tool calls to internal Elixir handlers or external REST APIs via HTTP bridge.

Quick Start

Get Steward ACS running in under a minute.

Terminal 
# Clone and deploy Steward ACS
git clone https://github.com/NaharEmet/steward_acs.git
cd steward_acs

# Generate secrets and configure
echo "SECRET_KEY_BASE=$(mix phx.gen.secret)" >> .env

# Launch with Docker

# Verify it's alive

Getting Started

Docker Quick-Start

The fastest way to get Steward ACS running is with Docker Compose. The project includes a pre-configured docker-compose.steward_acs.yml file that sets up the application with PostgreSQL.

Terminal 
# Clone the repository
git clone https://github.com/NaharEmet/steward_acs.git
cd steward_acs

# Generate a secret key base
echo "SECRET_KEY_BASE=$(mix phx.gen.secret)" >> .env

# Set the database URL
echo "DATABASE_URL=ecto://postgres:postgres@localhost:5432/acs" >> .env

# Start the stack
docker compose -f docker-compose.steward_acs.yml up -d

# Verify health
curl http://localhost:4001/health
# → {"status":"ok","version":"1.0.0"}

Production Setup

For production deployments, you'll need:

  • A PostgreSQL database (version 14+)
  • A secret key base for session signing
  • Appropriate environment variables (see Configuration Reference)
Production 
# Build release
MIX_ENV=prod mix distillery.release

# Run (or use systemd / container orchestration)
PORT=4001 \
  SECRET_KEY_BASE="your-secret" \
  DATABASE_URL="ecto://user:pass@host:5432/acs" \
  ACS_CLUSTER_NAME="production" \
  POOL_SIZE=20 \
  _build/prod/rel/steward_acs/bin/steward_acs start

Verifying the Installation

Once running, check these endpoints:

  • GET /health — Health check with version info
  • GET /api/tools — List all registered MCP tools
  • GET /api/presence — See active agents

Standard Workflow

Elixir 
# Before starting work — get context
generate_guidance_packet(scope_path: "my/area")
search_memories(query: "relevant topic")

# Step 1: Create and claim a task
create_work(agent_id: "MyAgent", title: "Implement feature X")
# → task_id: "abc-123"

claim_work(agent_id: "MyAgent", task_id: "abc-123")

# Step 2: Lock files before editing
lock_file(agent_id: "MyAgent", task_id: "abc-123", file_path: "lib/my_module.ex")

# Step 3: Do the work (edit files, run tests, etc.)
# ...

# Step 4: Save learnings for future agents
save_memory(
  kind: "learning",
  title: "Pattern: use guard clauses for edge cases",
  content: "...",
  scope_path: "my/area"
)

# Step 5: Release the task (also unlocks all files)
release_work(agent_id: "MyAgent", task_id: "abc-123")

# Step 6: Submit feedback to auto-generate memories
submit_task_feedback(
  task_id: "abc-123",
  agent_id: "MyAgent",
  learned_for_agents: "..."
)

Architecture

Steward ACS is designed as a modular, layered system optimized for AI agent coordination.

Steward ACS runs as a standalone Phoenix web application (port 4001). It exposes MCP tools (acs_* prefix) that AI agents call directly. The system uses ETS for fast in-memory caching with a PostgreSQL (production) or SQLite3 (development) database for persistence. A background sweeper handles auto-release of stale tasks and locks. The MCP Gateway routes agent tool calls to internal Elixir handlers or external REST APIs via HTTP bridge.

System Layers

1. MCP Tool Gateway

The outermost layer. All agent interactions flow through the MCP Gateway. Tool definitions are stored as YAML files and can be hot-reloaded without restarting the server. The gateway supports two handler types: internal Elixir modules (compiled) and external REST endpoints (HTTP bridge).

2. Core Engine

The central coordination logic, built with Elixir OTP. Key subsystems:

  • Task Manager — GenServer-based lifecycle management. Creates, claims, and releases work units. 10-minute auto-release timer prevents stuck tasks. Similar-task detection uses fuzzy matching on task titles.
  • Lock Manager — Distributed file locking via ETS. Prevents multi-agent edit conflicts. Locks auto-release after inactivity (tied to task lifecycle).
  • Memory Store — Vector database for semantic search. Stores "eternal truths" — patterns, decisions, warnings. Uses LLM-generated embeddings for similarity search.
  • Presence System — Real-time agent tracking. Reports current task, purpose, application, and component for every connected agent.
  • Error Registry — Persistent error traces with acknowledgment and resolution workflow. Create investigation tasks directly from error traces.

3. Persistence Layer

Three-tier storage strategy:

  • ETS (in-memory) — Fast access for locks, presence, and active tasks. Sub-millisecond reads.
  • PostgreSQL (production) — Full persistence for tasks, memories, errors, and configuration.
  • SQLite3 (development) — Zero-config database for local development.

4. Background Processes

  • Sweeper — Periodically scans for stale tasks and locks. Auto-releases after 10 minutes of inactivity.
  • Memory Auditor — Validates memory quality and flags low-quality entries.
  • Synthesis Engine — Clusters related claims into synthesized insights.

Multi-Cluster Architecture

Steward ACS supports multiple independent clusters. Each cluster is identified by ACS_CLUSTER_NAME and maintains its own isolated task, lock, and memory namespace. This enables separate environments for development, staging, and production — or independent clusters per team within the same organization.
